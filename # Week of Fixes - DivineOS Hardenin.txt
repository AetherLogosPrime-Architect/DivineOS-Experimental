## Topic 2: Logging Infrastructure + Error Handling

> **This is the most important topic.** Without proper logging, you can never debug your own tool. Without proper error handling, you never even know something is broken.

### Ticket 2.1: Create centralized logging and fix error handling

**Create `src/divineos/core/logging_setup.py`:**
- One file configures loguru for the entire project
- Writes to `~/.divineos/divineos.log` (rotating, with retention)
- Console output for INFO+, file output for DEBUG+
- Delete `agent_integration/logging_config.py` (the 6 stdlib loggers going nowhere)
- Every file uses `from loguru import logger` -- no more stdlib logging

**Fix the 102 blind except blocks:**
- This is NOT "change Exception to SpecificError everywhere." That's busywork.
- The real fix: decide per-catch whether to **propagate** or **swallow**.
- Rule of thumb: if a function returns `Optional[something]` and catches Exception to return None, that's a swallow. Ask yourself: "does the caller check for None? Does anyone ever know this failed?" If no -- propagate the error.
- Start with the worst offenders: `mcp_event_capture_server.py` (4 swallows), `mcp_integration.py` (10 swallows), `async_capture.py` (7 swallows)

**Fix the 4 Bandit try/except/pass blocks:**
- `try: ... except: pass` is never acceptable. At minimum log the error.

### Ticket 2.2: Fix linter errors using ruff --fix (safe) then manual pass

**IMPORTANT: understand --fix vs --unsafe-fixes before running anything.**

```powershell
# SAFE fixes (875 of them) -- formatting, trailing commas, import sorting
# These NEVER change behavior. Safe to run blind.
ruff check src\divineos --select ALL --fix

# UNSAFE fixes (186 of them) -- type annotation changes, code transforms
# These CAN change behavior. Review each one.
# DO NOT RUN: ruff check src\divineos --select ALL --fix --unsafe-fixes
# Instead, run per-rule to review:
ruff check src\divineos --select UP006 --fix --unsafe-fixes  # one rule at a time
```

**After safe fixes, the remaining ~930 manual errors break down as:**
- 102 BLE001 blind-except -- addressed in Ticket 2.1
- 73 PLR2004 magic-value-comparison -- replace magic numbers with named constants
- 66 TRY003 raise-vanilla-args -- use proper exception messages
- 58 E501 line-too-long -- break long lines
- 53 EM101 raw-string-in-exception -- move strings to variables
- 49 PLC0415 import-outside-top-level -- move imports to top of file
- 43 ANN201 missing-return-type -- add return type annotations
- 39 C901 complex-structure -- functions too complex, need splitting

**Do NOT try to fix all 930 in one go.** Fix by category. One ruff rule code at a time:

```powershell
# See only BLE001 errors:
ruff check src\divineos --select BLE001

# See only C901 (complexity) errors:
ruff check src\divineos --select C901

# Fix one category, commit, move to next
```

**Mypy --strict (138 errors):**
- Mostly missing return type annotations and untyped function definitions
- Fix AFTER ruff, since ruff's ANN fixes will reduce mypy count
- Run `mypy src\divineos --strict` to see what's left

**Pylint remaining (after ruff fixes):**
- W1203 (56x): f-strings in logging -- use `logger.info("msg %s", var)` not `logger.info(f"msg {var}")`
- The rest will mostly resolve once duplicates are deleted and error handling is fixed

## Topic 3: Use Your Own Tool + PowerShell Classroom

> This is the most important section. You can build forever, but if you never USE your own tool, you're just writing code that exists. Not code that works.

### Ticket 3.1: PowerShell + DivineOS daily driver classroom

**Why this matters:** Right now, your AI runs your tool for you. You type "test it" and Claude runs the commands. That means:
- You don't know if your tool actually works
- You don't know what the output looks like
- You don't know what breaks
- You can't demo it to anyone
- You can't find real UX problems

**You need to be able to open PowerShell and run your tool yourself.** Not your AI. YOU.

#### Session 1: PowerShell fundamentals (you + your AI, 30 min)

Have your AI teach you these basics IN POWERSHELL (not in a markdown file, actually do them):

```powershell
# Where am I?
Get-Location
cd C:\Users\santa\Desktop\open_tests\divnineosv2

# What's in here?
dir
dir src\divineos

# Activate the virtual environment
.\.venv\Scripts\Activate.ps1

# Now divineos CLI is available
divineos --help

# If no .venv exists yet:
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -e .
```

Actually TYPE these. Don't copy-paste. Don't let Claude run them for you.

#### Session 2: Use every DivineOS command (you, hands on keyboard, 30 min)

```powershell
# Initialize the ledger
divineos init

# Check it created the database
dir .divineos\

# Log a test event
divineos log --type "test_event" --actor "me" --payload '{"message": "hello from me, not my AI"}'

# See what's in the ledger
divineos events --limit 5

# Check stats
divineos stats

# Verify integrity
divineos verify

# Search
divineos search "hello"

# If you have a Claude Code JSONL file, ingest it:
divineos ingest path\to\your\chat.jsonl

# Build context from recent events
divineos context --n 10
```

**Write down what broke.** Write down what felt weird. Write down what output was confusing. THOSE are your real bugs. Not what pylint says -- what YOU experienced.

#### Session 3: Make DivineOS YOUR daily driver (ongoing)

After every Claude Code session:
1. Export/save your session JSONL
2. Run `divineos ingest <that file>`
3. Run `divineos stats` to see your ledger growing
4. Run `divineos verify` to confirm integrity
5. Before starting next session: `divineos context --n 20` to see what you did last time

If you do this for one week, you will find 10x more real bugs than any linter ever will. You'll find UX issues ("why does this output look like garbage?"), data issues ("why did it store that wrong?"), flow issues ("why do I have to run 3 commands for this?").

**The goal:** by end of week, you should be able to open PowerShell, activate your venv, and use your tool without asking your AI how. You should KNOW what every command does because you've run it yourself. That's when you stop vibe coding and start building something real.
