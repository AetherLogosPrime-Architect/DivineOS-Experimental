A single-file HTML calculator with a coherent aesthetic (dark theme, gold + purple + green glow), keyboard shortcuts wired end-to-end, standard arithmetic including parentheses and decimals, sanitization of the display characters back to JS operators before evaluation. It's ~90 lines total. It works. It looks good. ## What's working

**The aesthetic is coherent, not decorative.** The color scheme has intent — operators glow green, numbers glow purple, equals is the focal gold. The `pulseGlow` animation on the calculator container is a nice touch that doesn't overpower. Hover states shift meaningfully. Nothing is doing extra work for no reason.

**Keyboard support is fully wired.** Enter for equals (with `preventDefault` so the form doesn't do something unexpected), Escape for clear, Backspace for delete, digits and operators and parens all mapped. This is the class of polish that tells me the author actually used the calculator before shipping it, not just wrote it.

**Floating point noise is truncated.** `parseFloat(evalResult.toFixed(10)).toString()` handles the classic `0.1 + 0.2 = 0.30000000000000004` display problem gracefully without truncating meaningful precision. Small move, meaningful impact.

**Error handling is present at the right layer.** Infinity, -Infinity, NaN, and thrown SyntaxErrors all resolve to `'Error'` on the display. The user never sees a raw JS trace. Simple, correct.

**Self-contained single file.** No build step, no dependencies, no CDN pulls. Paste it into any browser and it runs. That's the right shape for a Discord artifact — matches the deployment surface.

**Code is readable.** Short function names (`append`, `clearAll`, `backspace`, `calculate`, `updateDisplay`), one job per function, no premature abstraction. A reader can hold the whole thing in their head after one pass.

## Where I'd push

Naming these plainly. Small things that would take five minutes each and lift the artifact.

**1. `Function()` as a controlled `eval` is technically fine here but worth flagging.**

The line `Function('"use strict"; return (' + sanitized + ')')()` is `eval` by another name — it constructs a function from a string that includes user input. For a local calculator running only client-side JS this isn't a security problem, because the only person "injecting" is the user themselves and it's their own browser. But if this ever gets embedded in a page that displays untrusted content, or if the same pattern shows up in a server-side context, it's remote code execution. Worth naming as "safe here because of the deployment surface, but the pattern is one to know when it stops being safe."

If your AI wanted to remove even the local risk, a small operator-parsing loop (shunting-yard or a recursive descent parser) is ~40 lines and gives real isolation. Not worth it for this artifact; worth knowing for the next one.

**2. After `=`, the expression stays and next input appends to it.**

If I type `5 + 3 =`, I see `8`. If I then type `2`, the display becomes `5 + 3 2` and calculating gives me... whatever `5 + 3 2` is (probably `Error` because the space isn't real, but the concatenation happens). The intuition most users have is "after equals, next digit starts a new expression." Small fix: on the `calculate` path, either clear the expression and treat the result as the new starting value, or set a flag so the next digit-input replaces rather than appends.

**3. No visible feedback for invalid intermediate states.**

If a user types `1.2.3`, they see `1.2.3` on the display and the calculator only complains when they hit `=`. Not a bug, but a small polish: catching malformed decimal in the `append` handler (e.g. reject a `.` when the current number-token already contains one) would give immediate feedback.

**4. No division-by-zero explicit branch.**

`5 / 0` currently resolves through `Infinity` and displays as `Error`. That works, but a specific message ("Cannot divide by zero" or similar) would be a nice touch. The generic Error hides what specifically went wrong.

**5. The keyboard handler doesn't check event target.**

If this HTML got embedded in a larger page with any text input, the global `keydown` handler would append digits to the calculator display every time the user typed in the other input. Not a concern for the standalone artifact, but a defensive `if (e.target.tagName === 'INPUT' || e.target.tagName === 'TEXTAREA') return;` at the top of the handler would future-proof it.

**6. Minor: `min-height: 100vh` on body — good — but the `flex` centering means on very narrow mobile viewports the calculator width (fixed 340px + padding) will overflow.**

Adding `max-width: 100%` on `.calculator` or wrapping the width in a media query would keep it mobile-friendly. Small.

## What I'd try next if I were your AI

- **Add history.** Small array of last N calculations, shown above the expression. Very small change, big UX lift.
- **Add memory buttons (MC, MR, M+, M-).** Classic calculator affordances, ~15 lines of code.
- **Add unit conversion or scientific mode as a toggle.** The visual space to do it exists.
- **Ship a keyboard-only mode where the visual dashboard is optional.** The keyboard support is already there; a screen-reader-friendly variant would be a nice extension.
- **One test file.** A tiny `.spec.html` that runs a handful of assertions (`assertEquals(calculate('2+2'), '4')`) and prints pass/fail. Turns the artifact from "trust me it works" into "here's the proof."
